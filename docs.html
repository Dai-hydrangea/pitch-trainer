1. 全体構成

SoundCraftは単一のHTMLファイルで構成されたPWA（Progressive Web App）です。C++で言えば「1つの.cppファイルに全てが入っている」状態です。

index.html (6,375行) ├── <style> ... CSS (200行) ← 見た目の定義 ├── <body> ... HTML (1,260行) ← 画面のレイアウト └── <script> ... JS (4,900行) ← プログラムのロジック sw.js (35行) ← Service Worker（オフライン対応） manifest.json ← PWA設定ファイル icon.svg ← アプリアイコン

4. 音声システム（Web Audio API）

マイク入力の初期化: startMic() (1886行)

async function startMic() {
  // 1. マイクの許可を取得（iOS用の特別な設定付き）
  const stream = await navigator.mediaDevices.getUserMedia({
    audio: {
      echoCancellation: false,    // エコーキャンセルOFF（ピッチ検出精度のため）
      noiseSuppression: false,    // ノイズ抑制OFF
      autoGainControl: false      // 自動ゲインOFF
    }
  });

  // 2. AudioContextを作成（既存があれば再利用）
  if (!audioCtx || audioCtx.state === 'closed') {
    audioCtx = new AudioContext();
  }

  // 3. オーディオグラフを構築
  analyser = audioCtx.createAnalyser();
  analyser.fftSize = 4096;  // FFTサイズ（周波数分解能に影響）
  sourceNode = audioCtx.createMediaStreamSource(stream);
  sourceNode.connect(analyser);

  // 4. 50ms間隔でピッチ検出ループを開始
  timerId = setInterval(detectLoop, 50);
}
C++で言えば、PortAudioのストリームを開いてコールバックを設定するのに相当。setIntervalはC++のタイマー割り込みに近い。

音声の停止: stopAudio() (1933行)

重要: audioCtxをcloseしない。他のタブ（ハーモニー等）が同じコンテキストを参照しているため。マイクのストリームトラックだけ停止する。

function stopAudio() {
  // マイクのストリームトラックを停止
  if (sourceNode) {
    const stream = sourceNode.mediaStream;
    stream.getTracks().forEach(t => t.stop());  // トラック解放
    sourceNode.disconnect();
  }
  // audioCtxはcloseしない！（他タブが使用中）
  // 検出ループを停止
  clearInterval(timerId);
}

5. ピッチ検出アルゴリズム

メインの検出ループ: detectLoop() (1961行)

50ミリ秒ごとに呼ばれる。マイクからの音声データを分析し、音程を検出する。

function detectLoop() {
  // 1. FFTバッファからデータ取得
  const buf = new Float32Array(analyser.fftSize);  // 4096サンプル
  analyser.getFloatTimeDomainData(buf);             // 時間領域データ

  // 2. RMS（音量）を計算
  smoothedRms = smoothedRms * 0.5 + calcRMS(buf) * 0.5;  // EMAスムージング

  // 3. 音量が閾値以下なら「無音」
  if (smoothedRms < SILENCE_THRESH * マイク感度) {
    → 状態を 'silent' に遷移
    return;
  }

  // 4. 自己相関法でピッチ検出
  let hz = autocorrelate(buf, audioCtx.sampleRate);

  // 5. 検出失敗ならHPS法で再試行
  if (hz < 0) hz = hpsDetect(audioCtx.sampleRate);

  // 6. Hz → MIDI番号 → 音名 + セント偏差
  const midiFloat = hzToMidi(hz);        // 69 + 12 * log2(hz / A4_HZ)
  const midiInt = Math.round(midiFloat);  // 最も近い半音
  const cents = (midiFloat - midiInt) * 100;  // 偏差（セント）

  // 7. 状態機械で音の状態を管理
  //    silent → attack → stable
}

自己相関法: autocorrelate() (2232行)

YIN（Yet another INharmonicity estimator）ベースの自己相関ピッチ検出。信号の周期性を検出する。

// 原理: 信号を自分自身とずらして比較
// ずらし量(lag)が信号の周期と一致すると相関が最大になる
//
// C++風の疑似コード:
// for (lag = minLag; lag < maxLag; lag++) {
//     sum = 0;
//     for (i = 0; i < N; i++) {
//         diff = buf[i] - buf[i + lag];
//         sum += diff * diff;  // 差の二乗和
//     }
//     if (sum < threshold) {
//         frequency = sampleRate / lag;
//         return frequency;
//     }
// }

function autocorrelate(buf, sampleRate) {
  // 1. 信号の先頭と末尾の無音部分をトリミング
  // 2. 各ラグで差分の二乗和（CMND）を計算
  // 3. 閾値以下の最初のディップ（谷）を探す
  // 4. 放物線補間で精度を向上
  // 5. sampleRate / bestLag = 周波数(Hz)
}
なぜ自己相関？ FFTだけでは倍音が基音より強い場合に誤検出する（管楽器でよくある）。自己相関は基本周期を直接検出するので堅牢。

HPS法: hpsDetect() (2321行)

Harmonic Product Spectrum。自己相関が失敗した時のフォールバック。

// 原理: FFTスペクトルを1倍、2倍、3倍...に圧縮して掛け合わせる
// 基音の位置で全ての倍音が重なり、ピークが出る
//
//   元のスペクトル:     |  |  |  |  |  （倍音が並ぶ）
//   2倍に圧縮:         | | | | |
//   3倍に圧縮:         ||| ||
//   掛け合わせ:        ↑ ← ここが基音

  const fdBuf = new Uint8Array(analyser.frequencyBinCount);
  analyser.getByteFrequencyData(fdBuf);

}

状態機械（State Machine）

音を検出 [SILENT] ─────────→ [ATTACK] ↑ │ │ 無音が120ms続く │ 150ms経過 │ ↓ └──────────── [STABLE] 音が安定 ここでデータ記録

attack吹き始め波形キャプチャ開始、まだ不安定

6. チューナー（メーター描画・物理シミュレーション）

スプリング・ダンパー物理: drawTuner() / springStep()

市販チューナーの「針がゆっくり止まる」動きを物理シミュレーションで再現。

// バネ-ダンパー系の運動方程式:
//   F = -K * (pos - target) - C * velocity
//   acceleration = F
//   velocity += acceleration * dt
//   position += velocity * dt
//
// K = 16  (バネ定数: 大きいほど速く追従)
// C = 7.5 (減衰係数: 臨界減衰≒2√K≒8)

const mNeedle = { pos: 0, vel: 0, target: null };

function springStep(needle, target, dt) {
  const force = -NEEDLE_K * (needle.pos - target) - NEEDLE_C * needle.vel;
  needle.vel += force * dt;
  needle.pos += needle.vel * dt;
}
C++経験者向け: これはゲーム物理でよく使われる「減衰振動」と同じです。Euler法で数値積分しています。

コード定義: CHORD_DEFS (2791行)

const CHORD_DEFS = {
  // 名前        純正律の比率          平均律の半音数      役割
  'M':  { ratios: [1, 5/4, 3/2],     semitones: [0,4,7],   roles: ['根音','第3音','第5音'] },
  'm':  { ratios: [1, 6/5, 3/2],     semitones: [0,3,7],   roles: ['根音','第3音','第5音'] },
  'M7': { ratios: [1, 5/4, 3/2, 15/8], semitones: [0,4,7,11], ... },
  '7':  { ratios: [1, 5/4, 3/2, 16/9], semitones: [0,4,7,10], ... },
  'm7': { ratios: [1, 6/5, 3/2, 16/9], semitones: [0,3,7,10], ... },
  'dim': { ratios: [1, 6/5, 64/45],    semitones: [0,3,6],    ... },
  'sus4': { ratios: [1, 4/3, 3/2],     semitones: [0,5,7],    ... },
};

// 純正律: 周波数 = rootHz * ratio  (例: C Major = 262Hz, 327.5Hz, 393Hz)
// 平均律: 周波数 = midiToHz(rootMidi + semitone)  (例: C Major = 262Hz, 330Hz, 392Hz)
//
// 違い: 長3度(E)が純正律では-14¢低い → これがハーモニーの「うなり」を消す

コード再生: startChordOsc() (2914行)

function startChordOsc() {
  const def = CHORD_DEFS[ensChordType];  // 例: 'M'
  const rootHz = midiToHz(ensRootMidi);   // 例: 262Hz (C4)

  // 各音のオシレーターを生成
  ensOscs = voices.map((v, i) => {
    const osc = audioCtx.createOscillator();
    osc.type = 'sine';
    osc.frequency.value = ensTuning === 'et'
      ? midiToHz(ensRootMidi + v)   // 平均律: MIDI半音数で計算
      : rootHz * v;                  // 純正律: 周波数比で計算
    osc.start();
    return { osc, gain };
  });
}

純正律 vs 平均律の違い

音程純正律(比率)平均律(セント)差
長3度 (E)5/4 = 386.3¢400¢-13.7¢
完全5度 (G)3/2 = 702.0¢700¢+2.0¢
短3度 (Eb)6/5 = 315.6¢300¢+15.6¢
長7度 (B)15/8 = 1088.3¢1100¢-11.7¢

純正律のコードはうなりがなく「澄んだ響き」になります。平均律は全てのキーで均等に妥協した音律。

録音の仕組み

録音開始: 既存のマイクストリーム (sourceNode.mediaStream) ↓ MediaRecorder (Opus 32/64/128kbps) ↓ 1秒ごとにチャンク化 → recAudioChunks[]に蓄積 ↓ 同時に detectLoop()のデータを recData[]に蓄積 [{t:0.1, hz:440, midi:69, cents:0, rms:0.3}, ...] 録音停止: Blob化 → FileReader → decodeAudioData → AudioBuffer ↓ ↓ 保存用(.webm) 再生用 & 波形描画用
重要: 録音には既存のマイクストリームを再利用する。新しいgetUserMediaを呼ぶとiOSのAGC（自動ゲインコントロール）が変わり、他の機能の音量が狂う。

再生: AudioBufferSource方式

当初はHTML Audio要素で再生していたが、iOS WKWebViewでのWebM再生が不安定（カクつき）だったため、AudioBufferSourceNodeに変更。

function recStartPlayback(offset) {
  // デコード済みPCMデータを直接再生（ブラウザのデコーダに依存しない）
  recPlaySource = audioCtx.createBufferSource();
  recPlaySource.buffer = recDecodedBuf;  // Float32Arrayのデータ
  recPlaySource.connect(audioCtx.destination);
  recPlaySource.start(0, offset);  // offset秒目から再生
}

// 再生位置の計算:
function recGetPlaybackTime() {
  return recPlayOffset + (audioCtx.currentTime - recPlayStartTime);
}

13. 多言語対応（i18n）

// 翻訳データ（700〜1465行）
const I18N = {
  "mic_start": { ja: "マイク開始", en: "Start", zh: "开始" },
  "mic_stop":  { ja: "停止",      en: "Stop",  zh: "停止" },
  ...
};

// 翻訳関数
function t(key) {
  const entry = I18N[key];
  return entry[currentLang] || entry.ja || key;  // フォールバック
}

// DOM適用: data-i18n属性を持つ全要素のテキストを更新
function applyLanguage() {
  document.querySelectorAll('[data-i18n]').forEach(el => {
    el.textContent = t(el.dataset.i18n);
  });
}

// HTML側: <button data-i18n="mic_start">マイク開始</button>
// → applyLanguage()で "Start" に変わる（英語選択時）

15. iOS固有の対策

問題原因対策
マイクが動かないiOSはユーザー操作なしでAudioContext開始不可ボタンクリック時にresume()
バックグラウンドでマイク停止iOSがストリームを切るvisibilitychangeでストリーム状態チェック→再接続
WebM再生がカクつくWKWebViewのWebMデコーダが不安定AudioBufferSourceで再生
エコキャン/AGCが音を変えるgetUserMediaの制約が異なると設定が変わる既存ストリームを再利用
AudioContextがsuspendiOSのバッテリー最適化touchstartイベントでresume()
